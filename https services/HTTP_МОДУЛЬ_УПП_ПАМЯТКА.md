# Памятка: модули HTTP-сервисов УПП

Эталонный код: **`bsl/ЗаказПоставщику/ЗаказПоставщику.bsl`**. Все новые модули HTTP-сервисов должны повторять его архитектуру и стиль (с учётом структуры обрабатываемых документов).

---

## 0. Структура каталогов модулей

Каждый модуль HTTP-сервиса размещается в **отдельном каталоге** внутри `bsl`, имя каталога совпадает с именем модуля (например, `ЗаказПоставщику`, `ТоварыНаСкладах`). Внутри каталога обязательно находятся:

- **Файл модуля** — `ИмяМодуля.bsl` (например, `ЗаказПоставщику.bsl`), содержащий экспортную функцию обработчика.
- **README** — файл `README.md` с описанием назначения сервиса, документацией по работе с параметрами запроса (таблица параметров, форматы, значения по умолчанию) и при необходимости пояснениями по формату ответа и отличиям от других версий.

Пример: `bsl/ЗаказПоставщику/ЗаказПоставщику.bsl` и `bsl/ЗаказПоставщику/README.md`. При добавлении или переписывании модуля создаётся каталог с таким именем, в него помещаются .bsl и README.

---

## 1. Шапка модуля (комментарии)

В начале файла — блок комментариев:

- Назначение: «Модуль HTTP-сервиса. Имя функции должно совпадать с именем шаблона URL в конфигураторе.»
- Кратко: что возвращает сервис (какие документы/данные).
- Структура JSON ответа (основные блоки, порядок полей).
- Список параметров запроса: перечисление через точку с запятой, в скобках — пояснение (например, `full (при true — полные блоки связей/ТЧ/вложений; иначе — сводки)`). Указывать, что по умолчанию возвращается базовый набор полей.

---

## 2. Сигнатура и общая обёртка

- **Имя функции** = имя шаблона URL в конфигураторе (например, `GetSupplierOrders` → шаблон `test` или тот, что указан в HTTP-сервисе).
- Один входящий параметр: `Запрос`.
- В начале: объявить массив результата и массив ошибок (например, `МассивЗаказов`, `МассивОшибок`).
- Весь код обернуть в **Попытка … Исключение**: при любом исключении — ответ 500, тело JSON `{"error": "текст ошибки"}`.

---

## 3. Разбор параметров запроса

**Принцип:** если суть параметра в разных модулях одна и та же (например, фильтр по номеру документа, по дате, расширенный ответ), то и **архитектура работы** с ним везде одинаковая: один и тот же порядок чтения, одни и те же форматы, одни и те же тексты ошибок и правила использования.

### 3.0. Единый порядок чтения параметров

Для **любого** параметра запроса используется один и тот же способ чтения (два шага):

1. Инициализация: `ЗначИмя = "";`
2. Чтение из тела/query: `Попытка ЗначИмя = СокрЛП(Строка(Запрос.ПараметрыЗапроса.Получить("имя"))); Исключение КонецПопытки;`
3. Запас из URL: `Если Не ЗначениеЗаполнено(ЗначИмя) Тогда Попытка ЗначИмя = СокрЛП(Строка(Запрос.ПараметрыURL["имя"])); Исключение КонецПопытки; КонецЕсли;`

Вложенная конструкция «Попытка Получить Исключение Попытка URL КонецПопытки КонецПопытки» **не используется** — везде только два шага выше. Так параметры читаются одинаково во всех модулях (code, name, from, to, date, warehouse, responsible, full и т.д.).

**Исключение для full:** после шагов 2–3 при пустом значении дополнительно проверять `ПараметрыURL["Full"]` и `ПараметрыURL["FULL"]` (из‑за разного регистра ключей в 1С).

### 3.0.1. Типы параметров и единая обработка

| Тип | Примеры параметров | Чтение | Использование |
|-----|---------------------|--------|----------------|
| **Строка (подстрока для ПОДОБНО)** | code, name, supplier/contractor, org, warehouse, responsible, comment, recipient, cash, status, contract, sender, receiver, inn, group, type, ngroup, article, material | По п. 3.0 (два шага) | Флаг `ИспользоватьФильтрИмя`; шаблон `"%" + Знач + "%"` (если не заполнено — `"%"`); в запросе `(НЕ &ИспользоватьФильтр ИЛИ Поле ПОДОБНО &Шаблон)`. При пост-фильтрации в БСЛ: шаблон с заменой %→* и _→?. |
| **Число** | summ, year | По п. 3.0 | `Число(Знач)` в Попытке; при ошибке — флаг не устанавливать (фильтр не применяется). |
| **Булево** | full, approved, service | По п. 3.0 (для full — плюс проверка Full, FULL) | Истина: `ВРег(Знач) = "1" Или "TRUE" Или "YES" Или "ДА"`. |
| **Даты** | from, to, date | См. п. 3.1 | Отдельный разбор форматов и КонецДня. |

- Источники: **сначала** `Запрос.ПараметрыЗапроса.Получить("имя")`, **запас** — `Запрос.ПараметрыURL["имя"]`.
- Обёртка в **Попытка … Исключение** при чтении (чтобы отсутствие параметра не ломало выполнение).
- Приведение к строке и отсечение пробелов: `СокрЛП(Строка(...))`.
- Для каждого параметра:
  - флаг использования фильтра: `ИспользоватьФильтрИмя = Ложь`;
  - переменная значения: по умолчанию пустая строка / дата 01.01.0001 / 0 и т.д.;
  - если `ЗначениеЗаполнено(ЗначПараметр)` — установить флаг и разобрать значение (даты — через `СтрРазделить` и `Дата(…)`, числа — `Число(...)`).
- Булевы параметры (`full`, `approved` и т.п.): считать «истиной» значения `ВРег(Знач) = "1"` Или `"TRUE"` Или `"YES"` Или `"ДА"`.
- Параметр **full**: в 1С ключи `ПараметрыURL` могут приходить с разным регистром — при чтении проверять по очереди `ПараметрыЗапроса.Получить("full")`, затем `ПараметрыURL["full"]`, `["Full"]`, `["FULL"]`.

### 3.1. Параметры дат (from, to, date) — единый подход во всех модулях

Во всех модулях, где есть фильтрация по датам, используется **один и тот же** способ чтения и разбора.

- **Источник**: сначала `ПараметрыЗапроса.Получить("from")` / `("to")` / `("date")`, при пустом — `ПараметрыURL["from"]` / `["to"]` / `["date"]` (два шага: Попытка из Запроса, Исключение КонецПопытки; затем Если Не ЗначениеЗаполнено — Попытка из URL, Исключение КонецПопытки; КонецЕсли).
- **Форматы даты**: поддерживаются **dd.MM.yyyy**, **dd-MM-yyyy**, **yyyy-MM-dd**. Разбор: `СтрРазделить(Знач, ".", Ложь)` и `СтрРазделить(Знач, "-", Ложь)`; при 3 частях через точку — Д=часть[0], М=часть[1], Г=часть[2]; при 3 частях через дефис — если длина первой части 4 символа, то Г,М,Д, иначе Д,М,Г. Дополнение нулями: Г до 4 знаков, М и Д до 2. Сборка: `Дата(Г + М + Д + "000000")`.
- **Период (from + to)**:
  - `ДатаНачала = Дата(Г1 + М1 + Д1 + "000000")` (начало дня);
  - `ДатаОкончания = КонецДня(Дата(Г2 + М2 + Д2 + "000000"))`;
  - в запросе: `(НЕ &ИспользоватьПериод ИЛИ (Поле.Дата >= &ДатаНачала И Поле.Дата <= &ДатаОкончания))`.
- **Одна дата (date)**:
  - `ДатаФильтр = Дата(Г + М + Д + "000000")`;
  - `ДатаФильтрКонец = КонецДня(ДатаФильтр)`;
  - в запросе: `(НЕ &ИспользоватьФильтрДата ИЛИ (Поле.Дата >= &ДатаФильтр И Поле.Дата <= &ДатаФильтрКонец))` (диапазон на весь день, не точное сравнение).
- **Ошибки**: при неверном формате — `МассивОшибок.Добавить("Некорректный формат from/to: " + ЗначFrom + " / " + ЗначTo)` или `"Некорректный параметр date: " + ЗначDate`; в блоке Исключение — `"Ошибка парсинга from/to: " + ...` или `"Ошибка парсинга date: " + ЗначDate`.
- **Имена переменных** в разборе даты: для периода — Г1, М1, Д1, ЧТ1, ЧД1 и Г2, М2, Д2, ЧТ2, ЧД2; для параметра date — Г, М, Д, ЧастиТочка, ЧастиДефис (единообразно во всех модулях).

Модули с параметрами дат: ЗаказПоставщику (from, to, date), ПоступлениеТоваровУслуг (from, to, date), ЗаявкаНаРасходованиеСредств (from, to, date), ТребованиеНакладная (from, to), ПеремещениеТоваров (from, to, date), ЗаказНаПроизводство (from, to), ТоварыНаСкладах (date — дата среза остатков).

## 4. Запрос к БД

- Текст запроса — многострочная строка с вертикальной чертой в начале строк (конструктор запроса).
- **Только параметризованные условия**: в ГДЕ использовать флаги и параметры, например  
  `(НЕ &ИспользоватьПериод ИЛИ (Поле.Дата >= &ДатаНачала И Поле.Дата <= &ДатаОкончания))`  
  и т.д.
- После текста запроса — явно вызвать `УстановитьПараметр` для **всех** используемых в запросе параметров.
- Для строковых «подобных» фильтров в SQL использовать шаблон с `%` (в 1С — `ПОДОБНО &НомерШаблон`); при необходимости пост-фильтрации в коде — отдельно переводить шаблон в вид для БСЛ (`%` → `*`, `_` → `?`).

## 5. Обработка выборки (цикл по результату запроса)

- **Пост-фильтрация** по строковым полям (если не заложено в запрос): через `СтрНайти(ВРег(Строка(Выборка.Поле)), ВРег(ПодстрокаПоиска)) <= 0 Тогда Продолжить`.
- Формирование одной записи ответа — через **Структура** и **Вставить**: явно добавлять каждое поле, чтобы порядок и наличие полей в JSON были предсказуемыми.
- Пустые/незаполненные значения:
  - строки реквизитов: подставлять текст вида «Не указан» / «Не указана» или пустую строку;
  - числа: при пустоте/неопределённости подставлять 0;
  - даты в выводе: всегда строка, например `Формат(Дата, "ДФ=dd.MM.yyyy")`; при отсутствии даты — безопасно проверять `ЗначениеЗаполнено` и при необходимости подставлять "".

---

## 6. Режим ответа (full)

- **По умолчанию** (без параметра full или full=false): всегда возвращается **базовый набор полей** документа (все основные реквизиты шапки: Номер, Дата, Контрагент, Организация, Ответственный, СуммаДокумента, Проведен, Комментарий и т.д. — по структуре документа). Плюс блоки связей и вложений в виде **сводок**: структуры с полями **Есть** (булево) и **Количество** (число), без выгрузки тяжёлых данных.
- **full=true**: к базовому набору добавляются **полные** данные: табличные части (если есть), полные списки связанных документов (с Номер, Дата, НомерДата), вложения с телом в Base64.
- Отдельный параметр **short** не используется: то, что раньше было «кратким» ответом, теперь является базой по умолчанию; для расширенной информации используется только флаг **full**.
- В коде: сначала формировать базовую структуру записи (все поля шапки), затем `Если ПолныйОтвет Тогда` — полные блоки (ТЧ, связи, вложения), `Иначе` — сводки по связям и вложениям (`Есть`, `Количество`).

---

## 7. Связанные документы и вложения

- Связи документов УПП уточнять по метаданным (в т.ч. через 1C-mcp): реквизиты основания, ТЧ, поля типа «Сделка», «ДокументОснование» и т.д.
- Уникальность в 1С — пара **Номер + Дата**; при формировании списков связей использовать ключ вида `Строка(Номер) + "|" + Формат(Дата, "ДФ=yyyyMMdd")` и дедупликацию через **Соответствие**.
- В JSON для каждой связанной записи при необходимости выводить поля **Номер**, **Дата**, **НомерДата** (например, «ИПЛ00000063 от 23.06.2025»); структуры формировать через **Вставить**, чтобы поля не терялись при сериализации.
- Вложения: при **full=true** — из **ХранилищеДополнительнойИнформации** по объекту документа; в ответе — массив элементов с **ИмяФайла** и **Данные** (Base64). Без full — запрос на **КОЛИЧЕСТВО** и в блоке по вложениям вывод **Есть** + **Количество**.

---

## 8. Итоговый ответ и ошибки

- Итоговая структура: `ИтоговыйРезультат` с ключами **data** (массив записей) и **errors** (массив строк ошибок).
- Сериализация: **ЗаписьJSON**, **УстановитьСтроку**, **ЗаписатьJSON**, **Закрыть**.
- Успех: **HTTPСервисОтвет(200)**, заголовки `Content-Type: application/json; charset=utf-8` и `Access-Control-Allow-Origin: *`, тело — полученная JSON-строка.
- Ошибка (внешнее Исключение): **HTTPСервисОтвет(500)**, тот же Content-Type, тело — JSON `{"error": "текст из ОписаниеОшибки()"}`.
- Ошибки по отдельным записям (например, по одному заказу) добавлять в `МассивОшибок`, не прерывая обработку остальных; в ответе они возвращаются в блоке **errors**.

---

## 9. Стиль и мелочи

- Имена переменных и параметров — осмысленные, на русском (как в эталоне): `ЗапросБД`, `ИспользоватьФильтрНомер`, `МассивПоступлений`.
- Запросы к БД выносить в переменные **Запрос** с суффиксом по смыслу (например, `ЗапросПоступ`, `ЗапросХДИ`); выполнение — **Выполнить()**, обход — **Выбрать()** и **Пока Выборка.Следующий()**.
- В блоках **Исключение** при обходе связей/вложений — добавлять в `МассивОшибок` понятный текст с номером/идентификатором записи и `ОписаниеОшибки()`.
- Перед добавлением записи в массив результата при необходимости выполнять «пробную» запись в JSON (как в эталоне), чтобы отсечь несериализуемые значения до отправки ответа.

## 11. Применимость к другим сервисам

- Для другого документа/справочника: та же схема (параметры → запрос → цикл → структура ответа → data/errors → 200/500).
- Набор фильтров и полей — под метаданные документа; флаг **full** — если есть тяжёлые данные (ТЧ, связи, вложения).
- Имя экспортной функции и шаблон URL — по конфигурации HTTP-сервиса УПП.

Эталон: **`bsl/ЗаказПоставщику/ЗаказПоставщику.bsl`** — при добавлении новых методов сверяться с ним по структуре и стилю. Документация по параметрам — в **`bsl/ИмяМодуля/README.md`**.
